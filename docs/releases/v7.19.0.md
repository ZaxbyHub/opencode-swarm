# v7.19.0

## What changed

- Added a native OpenCode plugin tool, `swarm_command`, for supported `/swarm` commands. The tool reuses the same command registry and handlers as the CLI path, so `agents`, `config`, `config doctor`, `status`, `show-plan`, `help`, `history`, `evidence`, `evidence summary`, `retrieve`, `diagnose`, `preflight`, `benchmark`, `doctor tools`, `knowledge`, `sync-plan`, and `export` now have one canonical implementation path.
- Reworked `command.execute.before` so chat-typed supported `/swarm` commands are routed to `swarm_command` for agents that own the tool. Commands outside the v1 tool allowlist keep the canonical in-hook fallback text with a verbatim-output instruction instead of pretending OpenCode can suppress the LLM response.
- Added policy gates around `swarm_command`: knowledge mutators are excluded, `config doctor --fix` is blocked, retrieve IDs and optional command arguments are validated, and the allowlist is enforced after command aliases resolve.
- Registered `swarm_command` through the tool index, plugin `tool` block, `TOOL_NAMES`, `TOOL_DESCRIPTIONS`, and `AGENT_TOOL_MAP`, including multi-swarm prefixed agent coverage through existing agent configuration generation.
- Added a system-transform rule for agents with `swarm_command` so the model calls the tool exactly once when the command hook injects a tool-call pointer and returns the tool output verbatim.

## Why

OpenCode currently runs `prompt(...)` unconditionally after `command.execute.before`, and hook return values are discarded. That means mutating `output.parts` alone cannot produce a deterministic slash-command response. Routing the supported read/display commands through an OpenCode-native tool gives the model a real callable surface while preserving future room to add commands through the shared registry and policy layer.

## Migration steps

No user migration required. Existing `/swarm` command registrations remain in place. Unsupported or state-changing commands still use the legacy chat-mediated fallback or the canonical CLI path until they receive explicit tool policies and confirmation gates.

## Breaking changes

None.

## Known caveats

- OpenCode still does not provide a plugin-side "no reply" flag for `command.execute.before`, so unsupported commands remain LLM-mediated in chat. Use `bunx opencode-swarm run <subcommand>` for canonical output when a command is not in the `swarm_command` v1 allowlist.
- The v1 tool intentionally excludes `knowledge migrate`, `knowledge quarantine`, and `knowledge restore` because they mutate persistent swarm knowledge and need confirmation semantics before being exposed to model tool calls.
